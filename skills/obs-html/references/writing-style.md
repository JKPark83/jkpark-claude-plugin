# 한국어 집필 스타일 (obs-html 4단계)

목표: **한국 사람이 읽었을 때 번역서 느낌이 전혀 나지 않는 글.**

## 1. 번역투 없애기

영어 자료를 옮길 때는 단어가 아니라 **의미**를 옮긴다.

| 번역투 ✗ | 자연스러운 한국어 ✓ |
|---------|-------------------|
| ~를 가진다 / ~을 가지고 있다 | ~가 있다, ~를 쓴다 |
| ~에 대해서 이야기해 보자 | 이제 ~를 살펴보자 |
| 그것은 ~하는 것을 가능하게 한다 | 이걸로 ~할 수 있다 |
| 우리는 ~할 필요가 있다 | ~해야 한다 |
| ~중의 하나이다 | 대표적인 ~다 |
| 이 함수는 값을 리턴한다 | 이 함수는 값을 돌려준다 / 반환한다 |
| ~되어질 수 있다 | ~할 수 있다 |
| 매우 높은 성능을 제공한다 | 성능이 뛰어나다 |
| 천장을 친다 (hits the ceiling) | 한계에 부딪힌다 |

다른 습관:

- **이중 피동 금지** — "보여지다"→"보이다", "사용되어진다"→"쓰인다"
- **무생물 주어를 사람/행위 주어로** — "이 설정은 캐시를 활성화한다" → "이 설정을 켜면 캐시가 동작한다"
- **"~에 있어서", "~을 통하여", "~로 하여금"** 은 쓰지 않는다
- 한 문장에 조사 "의"가 두 번 이상 나오면 다시 쓴다
- 문장은 짧게. 한 문단 3~5문장.
- 능동태 우선. 주어를 생략해도 뜻이 통하면 생략한다 (한국어의 기본).

## 2. 눈높이

### 기술 문서 — 주니어 개발자 기준

**아는 것으로 전제해도 되는 것**: 변수, 반복문, 함수, 클래스, HTTP 요청/응답,
git 기본, 터미널 사용.

**반드시 풀어줘야 하는 것**: 그 외 전부. 특히

- 처음 등장하는 전문 용어 — 한 줄 정의를 붙인다.
  > **멱등성(idempotency)** — 같은 요청을 몇 번 보내도 결과가 같다는 뜻이다.
- **왜 이게 문제인지** — 해법보다 문제를 먼저 설명한다.
- 비유. 추상 개념 하나에 일상 비유 하나.
- 코드는 붙여넣고 바로 돌아가는 최소 예제로. 핵심 줄에만 주석을 단다.

### 비기술 문서 — 전공자가 아니어도 이해되게

- 약어·업계 용어는 첫 등장 시 괄호로 풀어 쓴다.
- 추상적인 설명 뒤에는 반드시 **구체적인 예시** 한 개.
- 숫자는 맥락과 함께. "지연 200ms"보다 "지연 200ms — 눈으로 알아채기 시작하는 지점".
- 결론을 먼저, 근거를 나중에.

## 3. 구조

- 각 `h2` 섹션은 **한 가지**만 다룬다.
- 섹션 첫 문단에서 "이 섹션에서 뭘 알게 되는지"를 밝힌다.
- 주요 섹션마다 **그림 최소 1개** (obs-html 5단계).
- 핵심 주의사항은 `<blockquote>`로 눈에 띄게.
- 마지막 `한눈에 보기`는 불릿 3~5개.

## 4. 저장 전 자가 점검

- [ ] 소리 내어 읽어 걸리는 문장이 없다
- [ ] 이중 피동·"~를 가진다"·"~에 대해서" 가 하나도 없다
- [ ] 전문 용어는 전부 첫 등장에서 풀이했다
- [ ] 주요 섹션마다 그림이 있다
- [ ] 리서치에서 온 사실은 `참고 자료`에 링크가 있다
- [ ] 인터뷰에서 합의한 분량 범위 안에 있다
- [ ] `korean-reviewer` 지적사항을 반영했다
